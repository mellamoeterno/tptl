To use or not return:


My question:
would you say when i use find with an array function or array with objects, i'll always have to return before like in return menu.find() ?

Answer:

The point is actually not:

"Am I using find()?"

The point is:

"Do I want my function to return the result of find()?"

Case 1: Yes, use return
function getPizzaDetail(id: number) {
    return menu.find(pizza => pizza.id === id)
}

const pizza = getPizzaDetail(2)

getPizzaDetail() is supposed to give the caller a pizza, so you return the result of find().

Case 2: No, don't use return

Suppose you only want to log the pizza:

function printPizza(id: number) {
    const pizza = menu.find(pizza => pizza.id === id)
    console.log(pizza)
}

printPizza(2)

No return is needed because the function's job is to print something, not hand a value back.

Promise:

1. What’s inside the Promise constructor (logically speaking)
Technical description:
The Promise constructor takes a single argument called the executor function — a function that the constructor synchronously calls with two parameters: resolve and reject. Inside the executor, you typically start an async task, then call resolve(value) on success or reject(error) on failure.

“Whatever async work you do, itll use resolve to send back the successful result, or reject to send back the error.’ The constructor’s job is to store those resolve/reject controls and later connect them to .then() or .catch().”


2. When you use new Promise(...), what you’re expecting to get
Technical description:
new Promise(executor) returns a new Promise object in the pending state. That object has internal slots ([[PromiseState]], [[PromiseResult]], [[PromiseFulfillReactions]], etc.) and publicly exposes .then(), .catch(), and .finally() methods.

“When you write new Promise(...), you get back a special ‘future ticket’ object. That ticket starts in ‘waiting mode’ (pending). It doesn’t have the final value yet. What you can do right away is hand that ticket to someone else, who can attach reactions using .then() — those reactions will automatically run later when the ticket changes to ‘done’ (fulfilled) or ‘failed’ (rejected).”
